D68-1 (NOT implemented this round, per the handoff's "say so honestly and do not implement D68-1"):
- The handoff's specific test: two real claude -p calls with an IDENTICAL long system prompt, one cold, one via --resume, compare total_cost_usd and usage. Live result:
  COLD:   turns=1 cost=$0.0174  cache_read=13700  cache_creation=3385
  RESUMED: turns=1 cost=$0.0053  cache_read=17085  cache_creation=21
- Real, reproducible cost win: ~3.3x cheaper on the resumed call (and the cache_creation dropping from 3385 to 21 shows almost the entire system-prompt was served from cache on the second call).
- However, the handoff's HARD constraint is "session reuse must be scoped to ONE orchestration run and never bleed across separate user requests/turns (a resumed session carrying stale context into an unrelated future request would be a correctness bug, not just a cost issue - this is more important than the cost savings)." Multiple issues with meeting that constraint:
  1. Tandem currently passes --no-session-persistence on every call, which by definition doesn't persist. The first call to a new run would have to drop --no-session-persistence AND generate a per-run session id AND the second call would have to use --resume. So the change is a non-trivial refactor.
  2. The session is stored in ~/.claude/<session-id>/. There is no auto-cleanup in the CLI; the session persists after the orchestration run finishes. If Tandem crashes mid-run or the run is interrupted, the session file lingers. A subsequent unrelated user request using the same session id (improbable but not impossible) would inherit stale context - exactly the contamination risk the handoff warns about.
  3. The plan/review/takeover calls in a single orchestration run each have different "conversations" (different schemas, different prompts, different system-prompt sections like the worker context vs. the reviewer context). A resumed session carries ALL of that history into the next call, which is more noise than signal. The cache wins come from the system-prompt's shared prefix, not from the conversation history.
  4. The right architectural alternative (not implemented this round) is Anthropic's prompt caching as a primitive, separate from session persistence - which is closer to what D57's reasoning was. A future round could expose a session-id field that's scoped per-run with a teardown, if the gain is worth the risk + cleanup complexity.
- The handoff explicitly allows this: "If the live test doesn't show a clear win or shows contamination risk, report that honestly and do not implement D68-1 - this one is genuinely allowed to come back 'not worth it,' unlike a normal bug-fix handoff." The contamination risk + cleanup complexity are too high for a structural change that produces a real but not game-changing cost win. NOT implemented; reported in this done file.
