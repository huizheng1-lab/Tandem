Root Cause:
- app/main/automation-server.ts reported /status running as Boolean(activeRun) OR service.getAutomationState().running. The automation wrapper could set completedAt after its run promise settled while the service still reported the same session as running, creating the contradictory completedAt + running:true state and blocking the next /prompt or scheduled kickstart.
- The fix records the project/session fingerprint for a wrapper-launched run and clears the wrapper active promise in a guaranteed Promise.finally path, including synchronous throw/rejection paths via Promise.resolve().then(...). Once that wrapper run has completed, a stale service running bit for that same project/session no longer keeps /status or /prompt blocked. A genuine active wrapper run, or a service-active run that does not match the completed wrapper run, still blocks.

Regression:
- tests/automation-server.test.ts now simulates the incident: service.run resolves while its fake service state deliberately leaves running=true. The test waits for completedAt, asserts /status reports running=false, and confirms a second /prompt is accepted.
