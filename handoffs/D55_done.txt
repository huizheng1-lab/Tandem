D55-2 (shape-based heuristic, implemented with tight second-token check):

Before: hasCommandShape returned true only if the first token was in runnableCommandStarters OR a path prefix OR a script-filename pattern. An allowlist miss meant a hard reject.

After: hasCommandShape keeps the existing fast-paths, AND additionally accepts any token that (a) matches /^[A-Za-z][\w.-]*$/ (a single bare word starting with a letter) AND (b) either has a flag-shaped second token (`-x` or `--xxx`) OR the entry as a whole contains a path character (`/`, `\`, or `.`). Prose still fails because its second token is a plain word like "the" (not a flag) and its body has no path characters. Pure-prose regression tests confirm rejection.

Documented over-acceptance edge case: prose starting with verb-then-dash like "verify -all the -output" passes the heuristic because the second token "-all" matches the flag regex. This is exactly what the handoff describes as accepted ("the first token is followed by flags (`-x`/`--x`)"). A test locks in this current behavior so any future tightening is a deliberate test update, not a silent regression.

POSIX-tool guard (cat/grep/ls/sed/awk/head/tail/chmod) is UNTOUCHED. The check runs BEFORE hasCommandShape (separate code path), so POSIX tools still get rejected on win32 with the correct "use `findstr` instead" message. A regression test confirms grep/sed on win32 are still rejected.
