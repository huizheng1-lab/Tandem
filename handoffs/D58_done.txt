- Also caught and fixed a related bug in `dispatchStreams` (same commit, machine.ts:179-244). The original D54 dispatchStreams only spawned `poolSize` workers in a single initial loop and never refilled. With `poolSize=1` and `streams.length=2` (e.g. cap=1, two streams), only stream A would run, then the loop would exit with stream B silently skipped. The fix: each spawned promise has a `.then`/`.catch` that removes it from the in-flight set, and the outer while-loop refills any free slot. First-error capture also ensures a rejection still propagates so the build loop's try/catch converts it to a takeover. The D58 regression test caught this because the original D54 bug caused the second stream to be silently skipped, which would have hidden the carry-forward bug too.
