D56-1: destructive-format false positive on ffprobe/ffmpeg

Bug: src/tools/permissions.ts destructivePatterns included /\bformat\b/i. Tested directly:
  /\bformat\b/i.test('ffprobe -v error -print_format json -show_format "...tandem-explainer-en.mp4"') // true
That hard-blocked any ffprobe/ffmpeg call that used the extremely common `-show_entries format=duration` idiom, even with `permissionMode: "yolo"` (the destructive check at line 32 runs BEFORE the yolo bypass at line 35).

Fix: narrowed the pattern to require `format` followed by whitespace, optional Windows switches, and a drive-letter-shaped argument. The regex:
  /\bformat\s+(?:\/[a-z]+(?::[a-z]+)?\s+)*[a-z]:/i
- Matches: `format C:`, `format c:`, `format /FS:NTFS C:`, `format C: /FS:exFAT /Q`, `format a:`
- Does NOT match: `ffprobe -show_entries format=duration ...` (the bug-report failing case)
- Does NOT match: `ffmpeg -format json` (no whitespace, no drive letter)
- The other three patterns in destructivePatterns (rm -rf root, fork bomb, del /fsq C:\) are unchanged - audited via the D55 ffprobe/ffmpeg test set to confirm no other false positive.

Tests added (tests/tools.test.ts, 6 new in a "D56" describe):
- exact bug-report ffprobe command returns false.
- common ffprobe/format-duration idiom returns false (5 commands).
- common ffmpeg invocations return false (6 commands including ffmpeg -formats, ffmpeg -codecs, ffplay).
- format=... flag-style usage anywhere returns false (4 commands including `somebinary --format json`, `convert -format png`, `magick -format %w`).
- real disk-format commands still return true (6 commands: `format C:`, `format c:`, `format /FS:NTFS C:`, `format C: /FS:exFAT /Q`, `format D: /FS:FAT32 /V:STICK`, `format a:`).
- rm -rf root + del /f C:\ + fork bomb still return true; documented that `rm -rf /usr/local/bin` STILL matches the destructive gate (a known over-match in the existing pattern, out of scope for D56-1).

D56-2: verification-script-tampering detection

Problem: validateCompletionReport / enforceVerification only check that the plan's verification COMMAND STRING is reported verbatim and passing. They have no visibility into whether the SCRIPT that command invokes was modified to make a failing check pass. In takeover, the leader is both implementer and sole judge of its own final result, so the normal "reviewer checks diff" safety net doesn't apply.

The handoff explicitly says: "This can't be fully solved (any agent with write access to its own verification script can weaken it - genuine independence would require a second model judging takeover output, a bigger architectural change out of scope here)."

Concrete mitigation implemented (smallest reliable check):

In src/orchestrator/artifacts.ts, added extractReferencedScriptBasenames() and detectVerificationScriptTampering(plan, report). enforceVerification() now calls detectVerificationScriptTampering() and throws if:
  - A file referenced by any plan.verification command (e.g. `node verify-video.js` -> verify-video.js), AND
  - That file's basename appears in report.filesChanged, AND
  - The basename is NOT mentioned anywhere in report.deviationsFromPlan.

Error message: `Verification script edited without disclosure: <file>. Add an entry to deviationsFromPlan for each edited script before resubmitting.`

This applies to ALL completion reports (worker, leader, takeover) - a worker that silently weakens its own verification script also fails the check, not just takeover. Correct safety posture.

Reference detection handles:
- bare filename: `node verify-video.js`
- paths: `node "C:\\path\\to\\verify-video.js"`, `bash ./scripts/run.sh`
- quoted args: `node "verify-video.js"`
- multiple extensions: .js, .mjs, .cjs, .ts, .py, .sh, .cmd, .bat, .ps1, .exe
- Windows + POSIX path separators (basename extraction normalizes)

Tests added (tests/artifacts.test.ts, 6 new in a "D56-2" describe):
- accepts a report that does NOT touch a referenced verification script.
- rejects a report that touches a referenced verification script without declaring it.
- accepts a report that touches a referenced verification script when deviationsFromPlan mentions it.
- matches verification scripts by basename across path/separator styles.
- does not flag the original transcript's bug shape: verify-video.js + widened tolerance + all-passing + empty deviationsFromPlan is REJECTED.
- skips the check entirely when the plan has no script-suffixed verification commands (e.g. `npm test` + `ffprobe ...`).
